threading هي المكتبة الرسمية في Python لإنشاء خيوط التنفيذ المتعددة (Threads). كل مفهوم تطبيقي في مادة نظام التشغيل 2 يعتمد عليها مباشرةً:

  • ✓
    Thread() — كل صندوق بيع، كل مراقب، كل عملية في الخلفية هي Thread مستقل
  • ✓
    Lock() — Mutex الكلاسيكي لحماية المخزون من Race Condition
  • ✓
    RLock() — Reentrant Lock للعمليات المتداخلة التي تحتاج نفس القفل مرتين
  • ✓
    Semaphore(n) — يحدد عدد الخيوط المسموح لها بالدخول المتزامن للمورد
  • ✓
    Event() — إيقاف الخيوط بشكل نظيف عند إغلاق البرنامج (global_stop_event)
  • ✓
    Condition() — داخل RWLock لتنسيق القرّاء والكتّاب
  • ✓
    Barrier() — في سيناريو Stress Test لإطلاق كل الخيوط في لحظة واحدة
  • ✓
    enumerate() — لجمع قائمة جميع الخيوط النشطة في Task Manager

Thread Pool — معالجة الفواتير المتزامنة بشكل منظّم
▼

بدل إنشاء Thread جديد لكل فاتورة (مكلف ومفوضر)، يُنشئ ThreadPoolExecutor مجموعة ثابتة من الخيوط تُعاد استخدامها. كل طلب بيع يُرسَل للـ Pool وينتظر دوره.

لماذا Pool بدل Thread مباشر؟
إنشاء Thread جديد لكل عملية يستهلك ذاكرة ووقت. Pool يُعيد استخدام الخيوط المتاحة — أسرع وأكثر كفاءة وقابلاً للتحكم في الحد الأقصى للتزامن.
# core.py thread_pool = ThreadPoolExecutor( max_workers=8, thread_name_prefix="PoolWorker" ) # ui_windows.py — إرسال فاتورة للـ Pool thread_pool.submit(self._process_sale_thread, cart_copy, customer)

multiprocessing import multiprocessing Python stdlib
عمليات متوازية حقيقية تتجاوز GIL — تقرير المبيعات
▼
لماذا لا تكفي Threads للمهام الحسابية؟
Python لديها GIL (Global Interpreter Lock) — قفل يمنع أكثر من خيط من تنفيذ كود Python في نفس اللحظة. النتيجة: Threads مفيدة لـ I/O لكن لا تُسرّع الحسابات الثقيلة. multiprocessing يُنشئ عمليات مستقلة — كل عملية لها GIL خاصها — موازاة حقيقية!

asyncio import asyncio Python stdlib
تزامن بدون خيوط — الطلبات الأونلاين على خادم TCP
▼

asyncio يُتيح معالجة طلبات متعددة في نفس الوقت بـ Thread واحد فقط — عبر تبديل المهام أثناء انتظار I/O (await). هذا هو أساس سيرفرات الويب الحديثة.

الفرق بين asyncio و threading:
Threading = عدة عمال في نفس الوقت (موازاة حقيقية للـ I/O)
asyncio = عامل واحد يتبدّل بين المهام بذكاء (Concurrency بدون Threads)
# online_orders.py — خادم TCP غير متزامن async def handle_client(reader, writer): data = await reader.read(1024) await asyncio.sleep(0.3) # محاكاة معالجة I/O writer.write(response) await writer.drain() server = await asyncio.start_server( handle_client, "127.0.0.1", 8899 )

طابور آمن للخيوط — قلب نظام Producer-Consumer
▼

queue.Queue هي بنية بيانات Thread-safe تلقائياً — لا تحتاج Lock يدوي. تستخدم داخلياً Condition و Lock للحماية.

  • 1
    Producer-Consumer: Producers تضع طلبات في Queue — Consumers تأخذ منها
  • 2
    AutoRestockSystem: طلبات إعادة التخزين تمر عبر Queue بين Scanner و Processor
  • 3
    Online Orders: الطلبات الواردة من الخادم تُوضع في queue قبل المعالجة
# queue.Queue مع حجم محدد = يمنع overflow تلقائياً restock_queue = queue.Queue(maxsize=50) # Producer يضع — يتجمد إذا كانت ممتلئة restock_queue.put(item) # Consumer يأخذ — يتجمد إذا كانت فارغة item = restock_queue.get(timeout=5)

Priority Queue فعّالة O(log n) — الجدولة و Starvation
▼

heapq تُحوّل قائمة عادية إلى Min-Heap — دائماً العنصر الأصغر في القمة. نستخدمها لبناء Priority Queue مخصصة تدعم الأولويات والـ Aging.

# core.py — PriorityQueue للزبائن class PriorityQueue: def __init__(self): self._heap = [] self._lock = threading.Lock() def push(self, item): with self._lock: heapq.heappush(self._heap, item) def pop(self): with self._lock: return heapq.heappop(self._heap)

sqlite3 import sqlite3 Python stdlib
قاعدة البيانات الكاملة — منتجات، فواتير، مدفوعات، Audit Log
▼

sqlite3 قاعدة بيانات علائقية كاملة داخل ملف واحد — لا تحتاج خادم خارجي. المشروع يستخدمها بأسلوب احترافي يعكس مفاهيم OS مباشرةً.

  • ✓
    WAL Mode (Write-Ahead Logging): يسمح بالقراءة أثناء الكتابة — ضروري للخيوط المتزامنة. بدونه كل خيط ينتظر الآخر!
  • ✓
    Transactions + Rollback: إذا فشلت عملية البيع، تُلغى كلها. لا بيانات نصف مكتوبة.
  • ✓
    Indexes: على اسم المنتج ورقم الفاتورة — يُسرّع البحث من O(n) إلى O(log n)
  • ✓
    Audit Log: جدول يُسجّل كل عملية (Thread اسمه، قبل، بعد، توقيت، وضع التزامن)
# database.py — تفعيل WAL Mode conn = sqlite3.connect(DB_PATH) conn.execute("PRAGMA journal_mode=WAL") conn.execute("PRAGMA synchronous=NORMAL") # Transaction آمنة مع Rollback try: conn.execute("BEGIN TRANSACTION") conn.execute("UPDATE products SET stock=? WHERE id=?", (new_stock, pid)) conn.execute("INSERT INTO invoices ...", (...)) conn.execute("COMMIT") except: conn.rollback() # تراجع عن كل شيء!

# ui_windows.py — تحويل كلمة المرور قبل المقارنة pw_hash = hashlib.sha256( self.pass_entry.get().encode() ).hexdigest() # قاعدة البيانات لا تعرف الكلمة الأصلية — فقط الـ Hash c.execute("SELECT id FROM users WHERE username=? AND password=?", (user, pw_hash))

psutil (Process and System Utilities) تتيح قراءة معلومات النظام الحقيقية — استخدام CPU، استخدام الذاكرة، معلومات العمليات. بدونها لا يمكن رؤية استخدام النظام الفعلي من داخل Python.

لماذا خارجية وليست مدمجة؟
لأن الوصول لمعلومات النظام يختلف بين Windows/Linux/macOS. psutil توحّد هذا الوصول في واجهة واحدة تعمل على كل الأنظمة.
# ui_windows.py — قراءة CPU و RAM حياً import psutil cpu_pct = psutil.cpu_percent(interval=0.5) mem_pct = psutil.virtual_memory().percent # يُستخدم في تبويب الأداء لرسم الرسوم البيانية self.perf_cpu_history.append(cpu_pct) self.perf_mem_history.append(mem_pct)
المشروع يعمل بدونها أيضاً!
الكود يتحقق من وجودها: if HAS_PSUTIL: — إذا لم تكن مثبتة يظهر "N/A" بدل الرسم البياني.

معالجة الأخطاء وتتبعها في الخيوط
▼

traceback يُستخدم في Except blocks داخل الخيوط لطباعة تفاصيل الخطأ كاملاً. مهم لأن الخيوط لا تُظهر الأخطاء تلقائياً في الواجهة — يجب التقاطها يدوياً.

# داخل خيط — التقاط أي خطأ وعرضه import traceback try: # عملية البيع ... except Exception as e: err_msg = traceback.format_exc() self.after(0, lambda: self._log(f"خطأ: {err_msg}", "err"))
